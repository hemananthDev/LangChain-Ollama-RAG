# Mechanics

This document explains the underlying concepts and techniques that power LangChain-Ollama-RAG.

---

## What is RAG?

Retrieval-Augmented Generation (RAG) is a pattern that improves LLM answers by supplying relevant external knowledge at query time rather than relying solely on what the model learned during training.

Without RAG, an LLM can only answer from its training data — it knows nothing about your private documents. With RAG:

1. Your documents are pre-processed and stored as vector embeddings in a database
2. When you ask a question, the most relevant document chunks are retrieved
3. Those chunks are injected into the prompt as context
4. The LLM generates an answer grounded in that context

This means the LLM never needs to be retrained or fine-tuned to know about your documents.

---

## Embeddings

An embedding is a fixed-length list of floating-point numbers (a vector) that represents the semantic meaning of a piece of text. Two pieces of text with similar meaning will have vectors that are close together in the high-dimensional embedding space.

In this project, embeddings are generated by `nomic-embed-text:latest` running locally via Ollama. This model converts any text string into a 768-dimensional vector.

Embeddings are used in two places:

- **At ingestion time** — each document chunk is embedded and stored in ChromaDB
- **At query time** — the user's question is embedded and compared against stored chunk embeddings to find the most similar ones

The same embedding model must be used for both ingestion and retrieval, otherwise the vector spaces won't match. This is enforced by the single `EMBEDDING_MODEL` constant in `config.py`.

## Document Loading

LangChain provides loader classes that abstract the format-specific parsing of documents:

| Format | Loader class | What it does |
|---|---|---|
| `.txt` | `TextLoader` | Reads the file as plain UTF-8 text |
| `.pdf` | `PyPDFLoader` | Extracts text page by page using `pypdf` |
| `.docx` | `Docx2txtLoader` | Extracts text from Word documents using `docx2txt` |

Each loader returns a list of `Document` objects. A `Document` has two fields:
- `page_content` — the raw text string
- `metadata` — a dict with source information (file path, page number, etc.)

## Chunking

LLMs have a context window limit — they can only process a certain number of tokens at once. Storing entire documents as single embeddings also produces poor retrieval quality because a single vector can't capture the meaning of a long document precisely.

Chunking solves both problems by splitting documents into smaller, overlapping pieces before embedding.

This project uses `RecursiveCharacterTextSplitter`, which tries to split on natural boundaries in this order: paragraphs (`\n\n`), lines (`\n`), sentences (`. `), words (` `), and finally individual characters. It only falls back to a finer boundary if the chunk would exceed `CHUNK_SIZE`.

### Parameters

- `CHUNK_SIZE = 1000` — maximum number of characters per chunk
- `CHUNK_OVERLAP = 100` — how many characters from the end of one chunk are repeated at the start of the next

The overlap prevents a sentence or idea that straddles a chunk boundary from being lost. Without overlap, a retrieval query might miss relevant content that was split across two chunks.

### Example

Given a document of 2500 characters with `CHUNK_SIZE=1000` and `CHUNK_OVERLAP=100`:

```
Chunk 1: characters   0 – 1000
Chunk 2: characters 900 – 1900   (100-char overlap with chunk 1)
Chunk 3: characters 1800 – 2500  (100-char overlap with chunk 2)
```

## Vector Store — ChromaDB

ChromaDB is an open-source, embedded vector database. "Embedded" means it runs in the same process as your application — there is no separate server to start.

Each knowledge base is stored as a separate Chroma collection in its own subdirectory under `VectorDB/`. This isolation means:

- Switching knowledge bases loads a completely different vector space
- Deleting a knowledge base is as simple as deleting its folder
- Multiple knowledge bases can coexist without interfering with each other

At ingestion time, `Chroma.from_documents` takes the list of chunks and their embeddings and writes them to disk. At query time, `Chroma(persist_directory=...)` loads the existing database from disk without re-embedding anything.

## Retrieval

When a question is asked, the retriever:

1. Embeds the question using the same `EMBEDDING_MODEL`
2. Performs a cosine similarity search against all stored chunk embeddings
3. Returns the `TOP_K` most similar chunks as `Document` objects

`TOP_K = 3` means the 3 most relevant chunks are retrieved. Increasing this gives the LLM more context but also increases prompt size and response time.

The retriever is created from the vector store with:

```python
vectorstore.as_retriever(search_kwargs={"k": TOP_K})
```

---

## The Prompt

The retrieved chunks are formatted into a single context string (joined by double newlines) and injected into a `ChatPromptTemplate`:

```
Answer the question using only the provided context.

If the answer cannot be found in the context,
respond that the information is not available.

Context:
{context}

Question:
{question}

Provide a detailed answer.
```

Key design decisions in this prompt:

- "using only the provided context" — prevents the LLM from hallucinating answers from its training data
- "respond that the information is not available" — gives the LLM a graceful fallback instead of guessing
- "Provide a detailed answer" — encourages complete responses rather than one-liners

## The LCEL Chain

LangChain Expression Language (LCEL) uses the `|` pipe operator to compose steps into a chain. The chain in this project is:

```python
chain = (
    {
        "context": retriever | format_docs,
        "question": RunnablePassthrough(),
    }
    | prompt
    | llm
    | StrOutputParser()
)
```

Step by step when `chain.invoke(question)` is called:

1. The input `question` string is passed to both branches of the dict simultaneously
2. `retriever` fetches the TOP_K most similar chunks for the question
3. `format_docs` joins those chunks into a single context string
4. `RunnablePassthrough` passes the original question through unchanged
5. The dict `{"context": ..., "question": ...}` is passed to `prompt`, which fills the template
6. The filled prompt is passed to `llm` (ChatOllama), which generates a response
7. `StrOutputParser` extracts the plain text string from the LLM's response object

## Local Execution

Everything in this project runs locally:

- **Ollama** serves both the embedding model and the LLM as a local HTTP API on `localhost:11434`
- **ChromaDB** reads and writes to the local filesystem under `VectorDB/`
- **Streamlit** serves the UI on `localhost:8501`

No data is sent to any external service. No API keys are required.
